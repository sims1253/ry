## Part B: ecosystem regression net (scale up vendor_snapshot)

`crates/ry-checker/tests/vendor_snapshot.rs` currently snapshots diagnostics
over ONE vendored package (glue). Scale the mechanism so precision
regressions across many real packages are visible in PR diffs.

Design — a standalone harness rather than vendoring 50 packages into git:

1. New directory `ecosystem/` at the workspace root with:
   - `packages.txt`: list of package sources, one per line, format
     `name<TAB>git_url<TAB>pinned_ref`. Seed with ~15 pure-R, permissively
     licensed, widely used packages, pinned to release tags. Candidates
     (verify each is MIT/GPL-compatible-to-READ — any OSI license is fine
     since we only lint, we do not vendor): glue, rlang, purrr, dplyr,
     tidyr, stringr, forcats, withr, fs, cli, scales, lubridate, jsonlite,
     testthat, curl. (R source lives in each repo's `R/` directory.)
   - `run.sh`: clones/updates each pinned ref into `ecosystem/.cache/<name>`
     (shallow, `--depth 1 --branch <ref>` or fetch the ref), then runs the
     locally built `ry` (`cargo run --release -p ry-cli --`? check the
     actual package/bin name; prefer `cargo build --release` once then
     invoke the binary) with `--output-format` set to a machine format if
     one exists (check `crates/ry-checker/src/format.rs` for JSON/machine
     formats and the CLI flag name) over each package's `R/` dir, writing
     `ecosystem/reports/<name>.txt` as sorted `path:line:col CODE` lines
     (strip messages to keep diffs stable across message wording changes;
     keep a second `<name>.full.txt` with messages, gitignored).
   - `diff.sh`: runs `run.sh` twice — once against the merge-base build,
     once against HEAD — and prints a per-rule count diff plus the changed
     lines. SIMPLER ALTERNATIVE (preferred if merge-base builds are
     awkward): commit the `ecosystem/reports/*.txt` files to git as golden
     snapshots; `run.sh --check` regenerates and `git diff --stat`s them.
     A PR that changes diagnostic behavior then shows report diffs in its
     own diff. Choose this committed-snapshot approach unless you find a
     strong reason not to; it matches the existing insta-snapshot culture.
2. GitHub Actions job `ecosystem` (separate workflow, triggered on PR +
   manual dispatch): builds release binary, runs `run.sh --check`, fails if
   reports changed without being committed. Cache `ecosystem/.cache` and
   the cargo build dir with actions/cache.
3. Per-rule stats: `run.sh` also emits `ecosystem/reports/SUMMARY.md` — a
   table of rule code x package -> count, committed. This is the
   at-a-glance false-positive dashboard.
4. Keep runtime sane: 15 packages of pure R should check in well under a
   minute total given ry's speed; if any package is pathological, drop it
   and note why in packages.txt as a comment.

## Rules

- Read-only use of ../air and ../jarl.
- Network access: cloning the ecosystem packages requires network; if the
  sandbox has no network, still write all scripts + workflow and populate
  reports for packages already available locally (glue is vendored;
  ../air and ../jarl are NOT R packages, do not lint them), and state
  clearly in the summary that reports need one local `run.sh` execution.
- No mutating git commands (the run.sh clone commands operate on throwaway
  caches, that is fine; `git diff`/`rev-parse` read-only usage is fine).
- No emojis. Keep scripts POSIX-ish bash with `set -euo pipefail`.

## Acceptance criteria

- `dist-workspace.toml` (or equivalent) present and consistent with the
  reference repos' approach; release workflow present; README updated.
- `ecosystem/run.sh` works end-to-end locally (network permitting) and
  `--check` mode exits non-zero on report drift.
- Ecosystem workflow YAML valid; caching configured.
- SUMMARY.md generated with per-rule counts.
- `cargo test --workspace` untouched and green.
